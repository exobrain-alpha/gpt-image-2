# 项目目标

开发一个基于 Next.js 的本地文生图 Web 应用，接入 Microsoft Foundry 的 GPT-Image-2 模型。

# 技术要求

- 使用 Next.js
- 接入 Microsoft Foundry 的 GPT-Image-2
- 使用 TypeScript、TSX、Tailwind CSS
- 状态管理使用 Next.js 原生 Context

# 页面与功能要求

1. 页面必须包含四个核心模块：Prompt 输入框、生成结果预览区、历史生成结果列表、常用词或短语标签列表。
1. 页面整体分为三个区域：输入区、预览区、历史结果区。
1. 输入区包含 Prompt 输入框和常用词标签列表。
1. 输入区与预览区在同一行横向排列，二者平分可用宽度。
1. 历史结果区位于输入区和预览区下方，使用瀑布流布局展示历史生成结果。
1. 页面采用 body 滚动，无需额外的局部滚动容器。
1. 瀑布流列数需要根据浏览器视口宽度动态调整，单列最小宽度为 320px。
1. 需要适配不同设备尺寸，保证桌面端与移动端都有良好的使用体验。
1. 页面右下角提供返回顶部按钮；页面未发生滚动时不显示该按钮。

# 实现约束

- 避免产出超大文件
- 功能模块需保持清晰的模块化设计
- 不要偏离本文件定义的需求范围

# 调整&优化

1. 整体视觉

- 整个页面使用纯白背景。
- 所有主要内容直接放在页面上，不要使用卡片、面板、边框容器做分区。
- 不要添加多余说明文字、标题组、提示块、副标题、装饰性模块。
- 允许只给图片本身添加轻微阴影，其余区域保持克制和干净。

2. 页面结构

- 页面保留三个区域：左侧输入区、右侧当前预览区、下方历史结果区。
- 桌面端输入区和预览区横向排列，宽度大致各占一半。
- 历史结果区位于下方。
- 页面使用 body 自然滚动，不要做局部滚动容器。
- 移动端改为上下排列，并保证可用性和留白合理。

3. 输入区

- 输入区只保留以下内容：
  - 标题：Microsoft Foundry GPT-Image-2
  - 一个 prompt 输入框
  - 一个参考图选择入口
  - 一个参考图预览区域
- 参考图暂时最多只支持 1 张。
- 删除其余冗余控件和解释性内容。

4. 配置项标签化

- 将宽度、高度、质量、格式、常用词语全部统一改成标签式交互。
- 所有标签集中放在同一个区域中展示，不要拆成多个明显的分组面板。
- 标签点击后必须有明确高亮态，用于表示当前选中。
- 保持标签视觉统一，风格极简。
- 宽高、质量、格式是配置项标签，应支持切换当前值。
- 常用词语标签点击后应能参与 prompt 组织或直接填入 prompt。

5. 常用词语

- 常用词语不能写死为预设数据，必须支持用户自行创建。
- 当前阶段使用 JSON 文件做本地存储。
- 用户新增后，要能展示为标签。
- 点击常用词语标签后要高亮，并参与 prompt 逻辑。
- 交互保持轻量，不要做复杂管理界面。

6. 当前预览区

- 右侧当前生成结果区域直接展示图片，不要外层容器。
- 图片可带轻微阴影。
- 图片周围不要显示标题、prompt、参数说明、状态描述等文字。
- 若暂无图片，可保留极简空态，但不要做容器化设计。

7. 历史结果区

- 历史结果直接展示图片，不要使用卡片容器。
- 图片本身可带轻微阴影。
- 缩略图周围不要直接显示标题、prompt、参数等文字。
- 使用响应式瀑布流布局展示历史图片。

8. 历史图片详情

- 点击历史图片后，使用全屏灯箱查看。
- 灯箱中以图片为主。
- 右侧展示对应的 prompt 和其他必要信息。
- 信息布局保持简洁，不要添加复杂装饰。

9. 瀑布流修正

- 修正当前瀑布流分配逻辑，不要按固定顺序轮询插入。
- 必须改为：每次新增图片时，都插入到当前高度最短的列。
- 示例：
  - 3 列时，前 3 张图分别进入 3 列
  - 第 4 张图进入当前最短列
  - 后续每张图都进入当时最短列
- 该逻辑必须在不同屏宽和不同列数下都成立。

10. 实现限制

- 保持功能模块清晰，但不要为“模块感”增加卡片式视觉。
- 避免产出超大文件。
- 只做必要修改，不做额外发挥。
- 最终结果必须符合这几个关键词：极简、纯白、无容器、标签化配置、图片优先。
